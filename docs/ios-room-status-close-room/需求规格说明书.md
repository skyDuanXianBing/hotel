# 需求概述

本需求用于修复 iOS 端房态页中单房“关房/关闭房间”操作无法完成的问题。client 端当前可用实现已确认：单房和批量关房均调用 `/api/v1/room-status/blockouts/close`，请求参数为 `roomIds`、`startDate`、`endDate`、`type`、`remark`，成功后刷新房态日历。iOS 端已具备相同 API 封装和 store 方法，但房态首页单房快捷动作在打开关房弹窗时会被 Action Sheet dismiss 流程清空选中房间，导致提交时不发请求。

## 用户故事

作为酒店运营人员，我希望在 iOS 端房态页点击某一间空房后，可以直接选择“关房”，设置日期、关房类型和备注，并保存成功后在房态日历中看到该房间变为关房状态。

作为酒店运营人员，我希望在 iOS 端对已关房房间可以执行“开房”，恢复指定日期房间可售状态，并看到房态数据刷新。

作为酒店运营人员，我希望 iOS 端关房行为与 client 端一致，避免因为移动端参数或状态处理差异导致关房失败。

## 需求条目 (REQ-*)

| ID | 需求 |
| --- | --- |
| REQ-01 | 当用户在 iOS 房态首页点击某一间无订单且未关房的房间/日期并选择“关房”时，系统必须保留被选中的 `roomId` 和业务日期，直到关房弹窗提交或用户取消关房流程。 |
| REQ-02 | 当 iOS 提交单房关房时，系统必须调用 `roomStatusStore.closeRooms()`，并传入与 client 一致的参数结构：`roomIds: [roomId]`、`startDate`、`endDate`、`type: 'stop' | 'maintenance' | 'retain'`、可选 `remark`。 |
| REQ-03 | 当 iOS 单房关房弹窗没有有效房间上下文时，系统必须阻止提交并给出用户可理解提示，不能静默返回或让用户误以为已操作。 |
| REQ-04 | 当 iOS 单房关房或开房成功后，系统必须刷新房态日历和统计数据，并在当前日期格展示后端返回的 `closed`、`closeType`、`closeRemark` 状态。 |
| REQ-05 | iOS 修复范围必须限制在移动端房态交互层，优先允许修改 `ios/src/views/rooms/RoomsPage.vue`，必要时允许修改 `ios/src/components/room-status/CloseRoomModal.vue`；不得修改后端 API、client 端可用链路或无关页面。 |
| REQ-06 | 修复完成后必须至少在 `ios/` 下运行 `npm run build`；如增加或调整可单测的状态逻辑，应运行或补充 `npm run test:unit`。 |

## 功能列表 (Must Have / Should Have)

Must Have:

- 单房快捷动作选择“关房”后，关房弹窗能拿到正确房间信息，而不是显示“未选择房间”。
- 单房关房保存时实际发起 `POST /room-status/blockouts/close`。
- 单房关房成功后关闭弹窗并刷新房态，目标房间目标日期显示为关房状态。
- 已关房房间的“开房”继续可用，调用 `POST /room-status/blockouts/open` 并刷新房态。
- 对齐 client 的关房类型：`stop`、`maintenance`、`retain`。
- 不修改后端接口和 client 端现有可用实现。

Should Have:

- 关房弹窗提交前校验 `startDate <= endDate`。
- 当房间上下文缺失、日期缺失或后端返回失败时，iOS 给出清晰 Toast，而不是静默失败。
- 保持批量关房入口不回退；批量链路可继续使用 `BatchActionModal` 和 `roomStatusStore.closeRooms()`。

## 涉及的关键文件 (Context Outline)

### client 参考实现

- `client/src/views/room-status/RoomStatusCalendar.vue`
  - 单房快捷菜单中的“关房”入口调用 `handleQuickAction('close')`。
  - `handleQuickAction('close')` 设置 `closeRoomData`、`closeRoomForm` 并打开 `showCloseRoomDialog`。
  - `confirmCloseRoom()` 调用 `closeRoomBlockouts()`，成功后刷新 `loadRoomStatusCalendarData()`。
  - `confirmBatchCloseRoom()` 批量调用 `closeRoomBlockouts()` 或 `openRoomBlockouts()`。
- `client/src/api/roomStatus.ts`
  - 定义 `closeRoomBlockouts()`、`openRoomBlockouts()`、`DailyRoomStatusDTO.closed`、`closeType`、`closeRemark`。

### iOS 待修复范围

- `ios/src/views/rooms/RoomsPage.vue`
  - 房态首页主页面。
  - `RoomStatusCalendarGrid` 的 `open-room-actions` 事件进入 `handleOpenGridAction()`。
  - `openQuickAction()` 设置 `selectedRoomId`、`selectedBusinessDate` 并打开 `IonActionSheet`。
  - `quickActionButtons` 中“关房”调用 `handleOpenCloseRoom()`。
  - 当前缺陷点：`IonActionSheet @didDismiss` 触发 `handleQuickActionDismiss()`，清空 `selectedRoomId`，导致随后打开的 `CloseRoomModal` 失去 `selectedRoom`。
- `ios/src/components/room-status/CloseRoomModal.vue`
  - 单房关房弹窗。
  - 当前会在 `props.room` 为空时显示“未选择房间”，但保存按钮仍可点击并继续 emit。
  - 可增强为：无房间或日期非法时禁用保存/提示。
- `ios/src/stores/roomStatus.ts`
  - 已实现 `closeRooms()`、`openRooms()`、`getRoomBusinessState()`、`getRoomListItemById()`。
  - 原则上不需要大改；如修复需要移动“选中房间上下文”结构，可保持 store API 不变。
- `ios/src/api/roomStatus.ts`
  - 已与 client/后端参数一致，预期不需要修改。
- `ios/src/components/room-status/RoomStatusCalendarGrid.vue`
  - 点击空格或房间号格时发出 `{ roomId, date }`。
  - 当前事件绑定存在，问题不在网格点击事件缺失。
- `ios/src/views/room-status/RoomStatusDetailPage.vue`
  - 单房详情页已有“关房”按钮并调用 `roomStatusStore.closeRooms()`。
  - 可作为 iOS 端可用提交链路参考。

### 后端契约

- `server/src/main/java/server/demo/controller/RoomStatusController.java`
  - `POST /api/v1/room-status/blockouts/close`。
  - `POST /api/v1/room-status/blockouts/open`。
  - 均要求 `EDIT_ROOM_STATUS` 权限。
- `server/src/main/java/server/demo/service/RoomStatusService.java`
  - `closeRooms()` 校验日期、房间归属、权限、订单冲突，写入 `RoomBlockout` 并触发同步。
  - `openRooms()` 删除指定日期范围 `RoomBlockout`。
- `server/src/main/java/server/demo/dto/UpsertRoomBlockoutRequest.java`
  - 请求字段：`roomIds`、`startDate`、`endDate`、`type`、`remark`。
- `server/src/main/java/server/demo/enums/RoomBlockoutType.java`
  - UI 值仅支持 `stop`、`maintenance`、`retain`。

## 验收标准 (Checklist)

- [ ] REQ-01：在 iOS 房态首页点击空房格或房间号，选择“关房”后，关房弹窗显示正确房型与房号，不显示“未选择房间”。
- [ ] REQ-02：保存单房关房时，请求体包含正确 `roomIds`、`startDate`、`endDate`、`type`、`remark`，并调用 `/room-status/blockouts/close`。
- [ ] REQ-03：当房间上下文缺失或日期非法时，用户看到明确提示，且不会静默无响应。
- [ ] REQ-04：关房成功后房态刷新，目标日期显示为停用房/维修房/保留房；已关房房间执行开房后状态恢复。
- [ ] REQ-05：代码修改仅限 iOS 房态相关文件，不修改后端 API 与 client 可用实现。
- [ ] REQ-06：在 `ios/` 下 `npm run build` 通过；若调整或新增单测，`npm run test:unit` 通过。
