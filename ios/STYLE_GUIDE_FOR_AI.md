# iOS 样式改造限定规则

这份文档用于约束后续 AI 对 `ios/` 子项目的样式修改方式。

目标不是“自由发挥设计”，而是在当前首页已经确认过的视觉方向上，持续做一致、克制、可复用的样式演进。

适用范围：
- `ios/src/views/**`
- `ios/src/components/**`
- `ios/src/theme/**`

不适用范围：
- 接口、状态流、业务逻辑、路由结构
- 数据结构、计算逻辑、事件流程
- 任何与 UI 样式无关的重构

## 1. 总原则

后续 AI 在 `ios/` 做样式修改时，默认必须遵守以下规则：

1. 只改样式，不改接口、不改状态流、不改业务逻辑、不改路由。
2. 优先复用现有公共样式文件和 token，不允许每个页面重新发明一套视觉语言。
3. 页面视觉方向以当前首页为基准：
   - 浅蓝白背景
   - 轻卡片外层容器
   - 内部去卡片化
   - 强调信息层级，不强调花哨装饰
4. 能用公共 class 解决的，不新增页面私有重复样式。
5. 如果公共规则不适合某个页面，只允许做“最小必要调整”，并明确说明原因。
6. 禁止把页面做成重边框、重投影、重渐变、重装饰的“组件堆砌式”界面。
7. 禁止把一个页面内部嵌套成“外卡片 + 内卡片 + 小卡片”的多层卡片结构，除非该区域本身就是强信息块。

## 2. 当前公共样式文件

后续 AI 修改样式时，优先查看并复用以下文件：

- [tokens.css](D:/桌面/room/hotel/ios/src/theme/tokens.css)
- [patterns.css](D:/桌面/room/hotel/ios/src/theme/patterns.css)
- [ionic-overrides.css](D:/桌面/room/hotel/ios/src/theme/ionic-overrides.css)
- [variables.css](D:/桌面/room/hotel/ios/src/theme/variables.css)

职责划分必须保持清晰：

- `tokens.css`
  - 只放设计 token
  - 包括颜色、字号、字重、间距、圆角、阴影、基础语义变量
- `patterns.css`
  - 只放跨页面可复用的布局/组件模式类
  - 例如页面容器、卡片、hero、列表、pill、section header
- `ionic-overrides.css`
  - 只放 Ionic 原生组件的统一覆写
  - 例如 `ion-toolbar`、`ion-tab-bar`、`ion-button`、`ion-item`、`ion-input`
- `variables.css`
  - 作为统一入口，只做 import 汇总

不允许的做法：

- 把页面私有视觉规则直接堆进 `tokens.css`
- 把业务页面专属样式直接塞进 `patterns.css`
- 在多个页面里复制同一段样式而不是上提为公共 class

## 3. 颜色 Token 规则

当前颜色体系以 `tokens.css` 为准，后续必须优先复用，不要重新取色。

核心 token：

- 页面背景
  - `--ios-pms-bg-page`
  - `--ios-pms-bg-page-plain`
- 表层背景
  - `--ios-pms-surface`
  - `--ios-pms-surface-strong`
  - `--ios-pms-surface-muted`
- 文本颜色
  - `--ios-pms-text-primary`
  - `--ios-pms-text-secondary`
  - `--ios-pms-text-muted`
  - `--ios-pms-text-soft`
  - `--ios-pms-text-disabled`
- 边框与分隔线
  - `--ios-pms-border-soft`
  - `--ios-pms-border-faint`
  - `--ios-pms-divider`
- 主色与强调
  - `--ios-pms-primary`
  - `--ios-pms-primary-strong`
  - `--ios-pms-primary-soft`
  - `--ios-pms-primary-soft-active`

使用规则：

1. 页面级背景优先用 `--ios-pms-bg-page`，不要随意改成纯白或深色。
2. 大部分容器背景优先用 `--ios-pms-surface` 或 `--ios-pms-surface-strong`。
3. 正文标题统一用 `--ios-pms-text-primary`。
4. 描述文、说明文、辅助信息统一优先用 `--ios-pms-text-muted`。
5. 列表分隔线优先用 `--ios-pms-divider`，不要自己写更重的灰线。
6. 交互反馈优先使用主色浅底 `--ios-pms-primary-soft`，不要大面积实色铺满。
7. 危险态、成功态、警告态优先沿用 Ionic token，不单独发明一套红绿黄。

禁止：

- 新增高饱和大面积背景色
- 页面里同时出现多套无关主色
- 用深灰黑色重压整个页面
- 用过强描边替代层级设计

## 4. 字号层级规则

后续 AI 必须尽量使用现有字号 token，而不是页面里到处写死 `font-size`。

当前字号层级：

- 大标题
  - `--ios-pms-font-title-xl-size`
  - `--ios-pms-font-title-lg-size`
- 区块标题
  - `--ios-pms-font-title-md-size`
  - `--ios-pms-font-title-sm-size`
- 正文与说明
  - `--ios-pms-font-body-md-size`
  - `--ios-pms-font-body-sm-size`
  - `--ios-pms-font-note-size`
  - `--ios-pms-font-date-size`
- 数值
  - `--ios-pms-font-metric-lg-size`
  - `--ios-pms-font-metric-md-size`
  - `--ios-pms-font-metric-sm-size`

字重规则：

- 普通强调：`--ios-pms-weight-medium`
- 标题强调：`--ios-pms-weight-bold`
- 核心数字/重点标题：`--ios-pms-weight-heavy`

建议映射：

- 页面主标题：`title-lg` 或 `title-xl`
- 模块标题：`title-md`
- 列表项标题：`title-sm`
- 列表项描述：`body-sm`
- 辅助说明：`body-sm` 或 `note`
- 指标数字：`metric-lg`

禁止：

- 一个页面出现过多离散字号
- 用更粗更大的字替代层级结构
- 所有标题都做成同一尺寸

补充说明：

- 资料头部中的名称与邮箱不属于同一层级。名称可以接近 `title-sm` 到 `body-md` 之间的强调级别；邮箱、账号、辅助身份信息应降到 `body-sm` 或更轻，不要与主名称等重等大。
- 资料头部中的邮箱、账号类文本默认按单行处理，优先使用截断省略，不要因为长文本把头部撑成多行错位布局。

## 5. 页面背景规则

默认页面背景规则：

- 页面根容器优先使用 `.mobile-page`
- 仪表盘/首页类页面优先叠加 `.mobile-page--dashboard`
- 页面整体背景应与首页一致，保持轻、透、干净

适合：

- 首页
- 统计首页
- 设置首页
- 帮助/中心页
- 概览型入口页

不适合完全照搬的页面：

- 数据录入很重的表单页
- 长表格页
- 强操作页
- 弹窗型编辑页

对于不适合的页面，允许做的最小调整：

1. 背景仍保留浅体系，但可以弱化渐变感，改为 `--ios-pms-bg-page-plain`。
2. 容器密度可以更高，但不要改掉整体颜色体系。
3. 交互区可以更清晰，但不要转成后台管理系统那种硬边框风格。

## 6. 卡片样式规则

当前公共卡片模式：

- `.mobile-card`
- `.mobile-hero`
- `.pms-panel`
- `.pms-panel--hero`

统一要求：

- 外层容器有轻边框、圆角、柔和阴影
- 内层内容尽量去卡片化
- 卡片作为“区域边界”，不是“每个元素都单独成卡”

卡片基础特征：

- 边框：`1px solid var(--ios-pms-border-soft)`
- 圆角：`--ios-pms-radius-card` 或 `--ios-pms-radius-card-sm`
- 阴影：`--ios-pms-shadow-card`
- 背景：`--ios-pms-surface` 或 `--ios-pms-surface-strong`

适合：

- 首页模块
- 统计总览模块
- 设置分组模块
- 带标题的功能区块
- 账户资料头部的外层信息容器

不适合：

- 列表内部的每一行
- 已经在外层卡片里的次级信息块
- 本该只靠分隔线组织的信息区

例外处理：

如果某区域本身就是强数值块、图表块、核心 KPI 块，可以保留独立小卡片，但数量要少，层级要明确，不能泛滥。

补充说明：

- 设置首页、个人中心、门店资料页允许存在一个“资料头部 / 账户头部”容器，用于承载 logo/头像、名称、邮箱或辅助身份信息。
- 这类头部容器仍然属于轻表层卡片，不能做成营销横幅、会员大卡或高饱和品牌广告块。
- 资料头部优先使用“左侧头像或品牌图标容器 + 右侧两行文本”的基础结构，保持信息直接、清晰、自然对齐。
- 资料头部内部不要再嵌套次级卡片，头像/品牌图标容器可以是白底圆角矩形，但它只是局部承载，不是第二层大卡片。

不建议：

- 在设置首页头部叠加大面积品牌蓝渐变或权益卡式视觉
- 在资料头部加入与业务无关的夸张装饰块
- 把资料头部做成“头像卡 + 信息卡 + 操作卡”的三层结构

## 7. 按钮样式规则

按钮分两类处理：

1. Ionic 主按钮
2. 页面内轻操作按钮

统一原则：

- 主按钮用于关键操作
- 次级操作尽量轻，不做过强视觉噪音

主按钮建议：

- 继续走 `ion-button` 的统一覆写规则
- 使用主色，但不要叠加过重阴影
- 尺寸统一，不要单页特殊放大

轻按钮建议：

- 用浅底、轻边框或透明底
- 反馈态用 `--ios-pms-primary-soft`
- 如果是工具按钮，可使用轻卡片式按钮，但数量不宜过多

不允许：

- 同一屏出现多种完全不同风格的按钮
- 所有按钮都做成高饱和大色块
- 通过炫技按钮样式破坏页面信息重心

## 8. 列表项样式规则

当前推荐的列表项基础模式：

- `.pms-list`
- `.pms-list-item`
- `.pms-list-item__icon`
- `.pms-list-item__body`
- `.pms-list-item__title`
- `.pms-list-item__description`
- `.pms-list-item__arrow`

列表项必须遵守：

1. 列表结构优先采用“外层容器 + 行分隔线”。
2. 行与行之间优先使用顶部分隔线，不再每行加独立边框和圆角。
3. 列表图标区尺寸统一。
4. 标题和描述的字号、颜色、间距统一。
5. 列表项必须保持点击感，但不能过度卡片化。

适合：

- 设置入口列表
- 帮助中心列表
- 报表入口列表
- 快捷入口列表
- 设置首页中的账户相关入口列表

不适合直接照搬的场景：

- 需要左右双列信息对齐的账单行
- 高密度筛选器列表
- 复杂表单项

不适配时的最小处理：

1. 保留分隔线思路。
2. 保留标题/描述字体层级。
3. 只针对布局结构做调整，例如从三列改成两列或网格。

补充说明：

- 设置页列表项中的图标容器应轻量，优先使用浅底、轻边框、小圆角，不要把每个入口图标都做成高对比色块。
- 右箭头是提示性元素，颜色和权重应弱于标题与图标，不要成为页面视觉焦点。

## 9. 表单控件样式规则

Ionic 表单控件的样式统一通过 `ionic-overrides.css` 约束，不在页面里零散重写。

涉及控件：

- `ion-input`
- `ion-select`
- `ion-textarea`
- `ion-searchbar`
- `ion-segment`
- `ion-item`

统一要求：

1. 控件优先保持轻表层风格，不要做成厚边框输入框。
2. 圆角统一使用 `--ios-pms-radius-input`。
3. 输入区背景优先使用浅表层色，不要自己再叠一层重阴影。
4. 分段选择器应与整体页面颜色体系一致，不要局部跳色。
5. 表单页允许比首页更密，但仍需维持同一套 token。

适合：

- 设置中的主题切换
- 搜索输入
- 过滤条件
- 轻量录入页

不适合：

- 要求高度专业数据录入的复杂后台表单，如果遇到这类页面，可以提高信息密度，但不能脱离 token 体系。

## 10. 顶部栏与底部 Tab 规则

顶部栏规则：

- 统一通过 `ionic-overrides.css` 控制 `ion-header`、`ion-toolbar`、`ion-title`
- 顶部栏要轻、透、弱边界，不要厚重实体条
- 页面标题视觉重点更多应放在正文 hero 或页面首屏，而不是 toolbar 本身

底部 tab 规则：

- 统一通过 `ion-tab-bar`、`ion-tab-button` 的公共覆写管理
- 保持浅表层、轻阴影、明确选中态
- tab 数量要克制，避免重复入口

对后续 AI 的硬性要求：

1. 不允许随意新增 tab。
2. 不允许把其他页面已有功能重复放进 tab。
3. 若某 tab 移除，优先考虑迁移到首页快捷入口或设置页入口，而不是新建悬浮工具。

## 11. 间距规则

间距统一使用以下 token：

- `--ios-pms-space-1`
- `--ios-pms-space-2`
- `--ios-pms-space-3`
- `--ios-pms-space-4`
- `--ios-pms-space-5`
- `--ios-pms-space-6`

建议使用方式：

- 模块内微间距：`space-1` / `space-2`
- 行间距、列表间距：`space-3`
- 卡片内边距：`space-5`
- 模块与模块之间：`space-3` 到 `space-5`
- 大区块间距：`space-6`

统一要求：

1. 不要每个页面都写一堆 7px、11px、15px、19px 的离散值。
2. 同一页面的垂直节奏必须统一。
3. 信息密度高的页面可以收紧，但仍尽量落到 token 上。

## 12. 圆角与阴影规则

圆角 token：

- `--ios-pms-radius-input`
- `--ios-pms-radius-icon`
- `--ios-pms-radius-card-sm`
- `--ios-pms-radius-card`
- `--ios-pms-radius-pill`

阴影 token：

- `--ios-pms-shadow-card`
- `--ios-pms-shadow-card-strong`
- `--ios-pms-shadow-tabbar`

使用要求：

1. 卡片统一大圆角，不要页面里混入大量直角模块。
2. 图标容器统一小圆角。
3. pill 一律胶囊圆角。
4. 阴影只用于拉开层级，不用于制造厚重浮起感。
5. 页面内阴影数量要少，强度要轻。

禁止：

- 每个组件都加阴影
- 同一页出现多种互不相关的圆角尺度
- 用强阴影弥补布局问题

补充说明：

- 头像、品牌 logo、应用图标这类独立容器，允许使用介于 `radius-icon` 与 `radius-card-sm` 之间的中等圆角，但同一页面应保持一致，不要一处 18px、一处 24px、一处 30px 地混用。

## 13. 适合复用这套规则的页面类型

优先适用页面：

- 首页
- 概览页
- 统计首页
- 设置首页
- 功能入口汇总页
- 帮助中心页
- 轻量工具集合页
- 带少量表单的资料页

适用方式：

- 直接复用背景、卡片、分组标题、轻列表、pill、按钮规则
- 优先使用公共 token 和 `patterns.css` 里的模式类
- 如页面属于“资料页 / 账户页 / 设置首页”，可复用“资料头部 + 分组列表”的组合模式，但要保持轻表层，不得演变成营销卡片页

## 14. 不适合完全照搬的页面类型

以下页面不能生硬套首页风格，只能部分复用：

- 高密度表单页
- 复杂编辑页
- 房态/排班/日历类页面
- 表格型页面
- 多字段筛选页
- 强任务流页面
- 图表占主导的深度统计页

这些页面的处理原则：

1. 保留颜色 token、字体 token、圆角 token、阴影 token。
2. 保留浅背景和轻表层语义。
3. 不强行套用首页的 hero 或帮助中心列表结构。
4. 允许增加密度，但不要改成传统后台管理风格。
5. 若首页规则不适合，必须明确说明“不适合的点”和“做了哪些最小必要调整”。

## 15. 当页面不适配时，AI 必须这样处理

后续 AI 在回复中必须明确说明：

1. 哪些规则直接复用
2. 哪些规则不适合这个页面
3. 为什么不适合
4. 做了哪些最小必要调整
5. 是否只改了样式，没有动逻辑

标准表述建议：

- “本页直接复用了页面背景、卡片表层、列表分隔、pill、字号层级和间距 token。”
- “由于该页属于高密度录入页，不适合完整照搬首页帮助中心的去卡片化结构，因此只复用了颜色和表层规则，并对表单区域做了最小必要收紧。”

## 16. AI 修改样式时的执行顺序

以后让 AI 改 `ios/` 样式时，建议要求它按这个顺序执行：

1. 先判断页面类型
2. 再判断是否适合直接复用首页规则
3. 先复用 `tokens.css` 和 `patterns.css`
4. 只有公共样式不够时，才在页面内补充私有样式
5. 补充私有样式时，必须沿用 token，不得脱离体系
6. 修改完成后，明确说明“直接复用项”和“调整项”
7. 最后运行 `npm run build`

补充要求：

- 如果页面需要使用 logo、头像、插图等静态资源，优先放在 `ios/public/` 或项目内明确可引用的静态资源目录，不要直接引用 `dist/` 下的构建产物。
- 如果新增的是可复用品牌资源，应优先使用稳定命名，例如 `settings-logo.png`、`brand-mark.png`，避免使用带 hash 的临时文件名。

## 17. 给 AI 的固定约束模板

以后你可以直接把下面这段话发给 AI，作为样式修改前置条件：

```md
请基于 `ios/STYLE_GUIDE_FOR_AI.md` 执行本次样式修改，并严格遵守以下要求：

1. 只改样式，不改接口、不改状态流、不改业务逻辑、不改路由。
```

## 19. 任务流列表页补充规则

以下规则用于“审查页 / 审核队列页 / 待办列表页 / 轻量筛选结果页”这类页面。

这类页面的核心目标不是展示信息丰富度，而是让用户在手机上更快完成：

- 快速理解当前任务范围
- 快速筛选
- 快速扫读结果
- 快速进入详情继续处理

适用页面特征：

- 首屏只有少量标题、统计或工具入口
- 主体结构为“筛选器 + 结果列表”
- 列表项通常包含状态、时间、渠道、编号等多维信息
- 用户会高频重复进入详情页处理单条记录

统一原则：

1. 这类页面应优先做成“轻摘要 + 筛选面板 + 结果列表”的结构。
2. 首屏说明文案必须克制，避免把任务页做成介绍页。
3. 页面主焦点应放在筛选器和列表，不应让 hero、说明文字或工具按钮抢占视觉重心。
4. 列表优先强调扫读效率，而不是每条记录的独立卡片感。

### 19.1 Hero 使用规则

任务流列表页允许使用 `.mobile-hero`，但只应作为“轻摘要头部”，不应完整照搬首页 hero 的展示逻辑。

适合做法：

- 保留页面标题
- 可保留少量指标 chip
- 可保留一个次级工具按钮，如“链接列表”“导出”“筛选入口”
- 控制文案长度，避免首屏说明占据明显高度

不建议：

- 长段介绍性副标题
- 渐变、蒙层过重导致文字发虚
- hero 与下方主内容区权重接近，抢占任务页注意力

最小必要调整建议：

1. 允许对任务流页面单独弱化 `.mobile-hero` 的渐变和高光蒙层。
2. 允许删除静态副标题，只保留标题和关键指标。
3. hero 与下方 `.mobile-stack` 的纵向间距应与卡片之间的节奏一致，不要额外拉大或压得过紧。

### 19.2 筛选卡规则

任务流列表页的筛选器应视为“表单面板”，不是“内容区块”。

适合结构：

- 第一行：标题 + 筛选摘要，如“已筛选 x 项”
- 第二行：必要的单行说明
- 第三部分：筛选控件网格
- 最后一部分：操作按钮区

统一要求：

1. 筛选卡内部要有明确的头部 / 表单 / 操作区节奏。
2. 筛选控件默认优先使用两列网格；极窄屏再降为一列。
3. 同组 `ion-select`、`ion-input`、`ion-textarea` 的圆角、最小高度、label 节奏要统一。
4. 日期类控件与选择类控件应保持同一套视觉密度，不要出现单个控件明显更高或更松。
5. “搜索/应用”保留为主按钮，“清空/重置”必须更轻。

不建议：

- 在筛选卡内放多段说明文字
- 每个控件单独包一层小卡片
- 表单区和按钮区之间没有节奏分割
- 多个工具动作都使用主按钮权重

### 19.3 静态说明文案规则

对于任务流列表页中的静态说明文案：

1. 如果文案只是辅助说明，而不是错误、风险或动态反馈，优先按单行展示。
2. 单行展示时优先使用截断省略，不要因为静态文案把头部撑成两三行。
3. 在超小屏下允许回退成自动换行，避免极端截断影响理解。

适合单行处理的位置：

- hero 的静态说明
- 筛选卡的说明
- 结果栏的统计说明
- 资料头部的辅助身份信息

不适合单行强制处理的位置：

- 错误提示
- 空状态说明
- 加载提示
- 需要用户理解的操作风险提示

### 19.4 列表区规则

任务流列表页中的结果列表，优先采用“外层卡片 + 行分隔”的结构，而不是“每条记录一张卡”。

适合做法：

- 列表外层保留一个 `.mobile-card` 作为区域边界
- 内部记录项使用分隔线组织
- 第一行呈现主标题和状态
- 第二行呈现房间、渠道等次级信息
- 时间、编号等信息降到更弱层级

状态处理建议：

- 允许对待处理项做轻强调，但必须使用浅色背景或轻边框，不可用高饱和重色块
- 已处理项保持更克制，避免所有状态同等显眼
- 状态标签统一做成 pill，不要每种状态都做一套完全不同的按钮样式

不建议：

- 每条列表项都使用完整白卡、边框、圆角、阴影
- 为了强调某条记录，直接用负 margin 把条目做出“突破容器”的效果
- 用多个彩色块同时强调状态、编号、时间和操作

### 19.5 Flex 与溢出约束

任务流列表页中，标题区、筛选头部、结果栏、列表项头部经常采用 `display: flex`。

后续 AI 在这类页面中必须默认检查以下问题：

1. 左侧文本容器如果可能出现长标题、长说明或单行省略，必须显式设置 `min-width: 0`。
2. 右侧按钮组、pill、状态标签如果需要固定宽度或禁止压缩，应设置 `flex-shrink: 0`。
3. 只要出现“左边对齐正常、右边看起来外扩”的情况，先检查 flex 子项收缩，而不是直接怀疑卡片宽度。
4. 禁止通过负 margin 制造“强调条目超出容器”的视觉效果；这很容易造成右边界看起来不齐。

优先排查点：

- `.mobile-inline-row`
- hero 头部区域
- 筛选卡头部
- 列表头部
- 列表项标题区

### 19.6 适用性说明

本组规则直接适用于：

- 审查页
- 审核队列页
- 订单待处理页
- 轻量化任务列表页
- 带少量筛选的结果页

本组规则不适合完全照搬到：

- 高密度专业录入表单
- 表格页
- 图表主导型统计页
- 大量字段编辑页

如果页面不适合完整使用本组规则，应至少保留：

1. 浅表层背景和 token 体系
2. 筛选器与列表之间的明确层级
3. 头部 flex 收缩约束
4. 列表去卡片化思路
